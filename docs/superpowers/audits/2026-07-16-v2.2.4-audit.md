## Confirmed findings

### F-1. `MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` not bumped to `2.2.4`

- **Severity:** **BLOCKER** for v2.2.4 release.
- **Locations:**
  - `project.yml:12` — `MARKETING_VERSION: "2.2.2"`
  - `project.yml:13` — `CURRENT_PROJECT_VERSION: "2.2.2"`
  - `ClipMemory.xcodeproj/project.pbxproj:485` — `CURRENT_PROJECT_VERSION = 2.2.2;` (Release config)
  - `ClipMemory.xcodeproj/project.pbxproj:499` — `MARKETING_VERSION = 2.2.2;` (Release config)
  - `ClipMemory.xcodeproj/project.pbxproj:560` — `CURRENT_PROJECT_VERSION = 2.2.2;` (Debug config)
  - `ClipMemory.xcodeproj/project.pbxproj:580` — `MARKETING_VERSION = 2.2.2;` (Debug config)
- **Evidence:** `grep -RInE 'MARKETING_VERSION|CURRENT_PROJECT_VERSION' project.yml ClipMemory.xcodeproj/project.pbxproj …` returned the literal `2.2.2` at every line above.
- **Why it matters:** `xcodebuild -configuration Release build` will produce a `.app` whose `CFBundleShortVersionString`/`CFBundleVersion` still reports `2.2.2`, so `Releases/ClipMemory.tar.gz` and any subsequent `Homebrew/ClipMemory.tar.gz` artifact will ship an app whose version stamp contradicts the new release tag.
- **Reproduction:** `xcodebuild -project ClipMemory.xcodeproj -scheme ClipMemory -configuration Release -showBuildSettings 2>/dev/null | grep -E 'MARKETING_VERSION|CURRENT_PROJECT_VERSION'` → currently prints `2.2.2`.
- **Intended regression test:** unit test that reads the built `Info.plist` of a Release artifact and asserts `CFBundleShortVersionString == "2.2.4"` and `CFBundleVersion == "2.2.4"`.

### F-2. Homebrew cask `version` and `sha256` not bumped to `2.2.4`

- **Severity:** **BLOCKER** (Homebrew tap cannot install v2.2.4 if cask is stale).
- **Location:** `Casks/clipmemory.rb:2-3`
  - `version "2.2.0"`
  - `sha256 "8b8968d86ddf5610304e81e7795d33536cee0c933159b360e1b81226458063fc"`
- **Evidence:** direct `cat Casks/clipmemory.rb`.
- **Why it matters:** The cask URL pulls `https://github.com/irykelee/clipmemory/releases/download/v#{version}/ClipMemory.tar.gz`, which would resolve to a 404 (see R-3). The `sha256` is also stale; an updated cask must be computed after the new tarball is produced.
- **Reproduction:** `brew info --cask clipmemory` (would currently show `2.2.0`).
- **Intended regression test:** script-level check that asserts the cask `version` matches the latest git tag and that `sha256` matches `shasum -a 256 Releases/ClipMemory.tar.gz`.

### F-3. `Scripts/package.sh` default version `2.0.0`

- **Severity:** **HIGH** (silent footgun — easy to publish a `2.0.0`-stamped tarball).
- **Location:** `Scripts/package.sh:3` — `VERSION=${1:-2.0.0}`
- **Evidence:** direct `cat Scripts/package.sh`.
- **Why it matters:** If `Scripts/package.sh` is invoked without an explicit version argument, it packages `ClipMemory v2.0.0`, regardless of the actual `MARKETING_VERSION` in `project.yml`. The artifact filename `ClipMemory.tar.gz` is constant, so the tarball can be silently overwritten with an old version stamp.
- **Reproduction:** `./Scripts/package.sh` (no args) → echo line prints `Packaging ClipMemory v2.0.0...`.
- **Intended regression test:** shell test that asserts `${1:-...}` matches `MARKETING_VERSION` parsed from `project.yml`.

- **Severity:** **MEDIUM** (process/audit-trail concern; the same mistake risks repeating in v2.2.4).
- **Location:** commit `a57cdff` (tag `v2.2.3`), subject: `fix: address 3 v2.2.2 review findings (C1 + HIGH-1 + version)`.
- **Evidence:** `git show --no-patch --format='%H %s' v2.2.3` followed by inspecting `project.yml` at that commit (`git show a57cdff:project.yml` would confirm — recorded here from the current tree's identical state because `a57cdff` is an ancestor and the source-of-truth for these settings is unchanged post-v2.2.3 except for tests; see Verification log).

### R-3. GitHub release `v2.2.3` has zero assets

- **Severity:** **HIGH** (blocks install via Homebrew and direct download; documented because it is part of the v2.2.4 audit baseline even though it concerns the previous release).
- **Location:** `https://github.com/irykelee/clipmemory/releases/tag/v2.2.3`.
- **Evidence:** `gh release view v2.2.3 --json assets` returned `"assets": []`. The release was published `2026-06-30T07:03:04Z` but no `ClipMemory.tar.gz` was uploaded.
- **Why it matters:** `Casks/clipmemory.rb` (F-2) downloads `https://github.com/irykelee/clipmemory/releases/download/v#{version}/ClipMemory.tar.gz`. With no asset at that path, `brew install clipmemory` (or any direct download) 404s. This is the underlying reason F-2 is a hard blocker even if the cask string were updated.
- **Reproduction:** `gh release view v2.2.3 --json assets` → `[]`. Web check: visit `https://github.com/irykelee/clipmemory/releases/download/v2.2.3/ClipMemory.tar.gz`.
- **Intended regression test:** release CI step that uploads `ClipMemory.tar.gz` after `Scripts/package.sh` runs and that `gh release view <tag> --json assets` is non-empty before tagging the next release.

### Tags finding — duplicate `v2.1.0` in tag list

- **Severity:** **LOW** (cosmetic, but worth noting before tag-annotation work).
- **Location:** `git tag --sort=-creatordate` output includes `v2.1.0` twice.
- **Evidence:** `git tag --sort=-creatordate | head -12` shows `v2.1.0` on two consecutive lines.
- **Why it matters:** A duplicate tag points to two distinct commits with the same name — `git describe` can resolve to either depending on ref resolution rules. Cosmetic only for v2.2.4, but should be cleaned up before any future tag-pinning release tooling is built.
- **Reproduction:** `git show-ref --tags | grep v2.1.0` → two entries.
- **Intended regression test:** script that asserts `git tag | sort | uniq -d` is empty before cutting a new tag.

### Rejected suspicions (with source path and disproving evidence)

Per the brief: "For each rejected suspicion, record the source path and the existing test or code invariant that disproves it. Do not silently discard a suspected security issue."

- **RS-1 — `FileStorageBackend` async dispatch race:** `FileStorageBackend.save` (`ClipMemory/Services/StorageBackend.swift:31-36`) dispatches `UserDefaults.standard.set` to `DispatchQueue.main.async`. If the process exits before the dispatch fires, the write is lost. Disproven by: `testRestartRecoversItemsFromBackend` (IntegrationTests.swift:51-71) and `testDeleteItemPersistsAfterRestart` (109-118) exercise `MemoryStorageBackend` (sync), not `FileStorageBackend` (async), so the dispatch race is structurally untested. However, `UserDefaults` is buffered by `cfprefsd` which flushes on graceful app exit; the race window only matters on `Terminate()`-without-graceful-shutdown, which is a pre-existing design choice not introduced by v2.2.4. **Test gap, not a code defect.**
- **RS-2 — Cache pollution after `deleteAllExcept`:** `deleteAllExcept` (`ImageStorage.swift:268-276`) removes files from disk but does not clear `imageCache`. Disproven by: `deleteAllExcept` is only called from `cleanupOrphanedImages` (line 288), which only runs once per app launch from `ClipboardStore.loadItems()` (`ClipboardStore.swift:229`), at which point the NSCache is empty (in-memory only, resets on launch). After init, bulk deletion routes through `deleteItems` → per-file `deleteImage` (which DOES clear cache, line 263). `testCleanupOrphanedImagesRemovesUnreferenced` (ImageStorageTests.swift:346-365) covers the disk-state behavior; cache-state is not explicitly tested but timing makes pollution impossible in practice. **Theoretical gap, not a real risk.**
- **RS-3 — `constantTimeCompare` leaks length:** `CryptoService.swift:240` early-returns on `a.count != b.count`, which is non-constant-time with respect to length. Disproven by: both HMAC-SHA256 outputs are always 32 bytes (SHA-256 block is 64, output 32), so the early-return branch is unreachable in any production caller. Covered by `testConstantTimeCompareDifferentLengthsReturnsFalse` (CryptoServiceTests.swift:166-172). **Not exploitable.**
- **RS-4 — `addItem` encryption-failure path is unregressed:** `ClipboardStore.swift:280-299` discards the entire item when `encrypt()` returns nil. Disproven by: `encrypt()` returning nil requires either (a) `generateKey()` having thrown (which `fatalError`s inside `generateKey`, line 44 or 58), or (b) `AES.GCM.seal` throwing — extraordinarily rare for in-process data and never observed. The N2 path is provably correct by code reading; failure would have to cross `return` AND `NotificationCenter.post`. No regression test mocks this path. **Test gap; could be regressed by reintroducing `MockCryptoService`, but the most recent refactor (commit `4e832ce` "refactor(tests): 删除 MockCryptoService 死代码") intentionally removed it. Adding back for this one test would regress the refactor.**
- **RS-5 — `ImageStorage.migrateFromLegacyIfNeeded` flag-set-after-loop race:** `ImageStorage.swift:43-112` sets `migrationCompleteKey = true` after the per-file loop completes. If the loop raises an uncaught throw (e.g., OOM), the flag stays unset and the next launch re-attempts. Disproven by: each per-file body uses `try?` and `do/catch` (lines 53, 63, 76-83, 87-92); per-file errors do not abort the loop. Re-attempts are safe because the `isUnencryptedPNG` check (66-69) prevents double-encryption: already-migrated files don't match the PNG-signature and are simply copied (86-93). The migration code path itself is **uncovered by tests** — all `ImageStorageTests` set `ImageStorageMigrationComplete = true` in `setUp` (ImageStorageTests.swift:36-37) so the migration is bypassed. **Test gap, not a code defect.**
- **RS-6 — Legacy-format AES-CBC + HMAC direct round-trip is unexecuted in tests:** `decryptLegacy` (`CryptoService.swift:179-210`) and `legacyDecryptImage` (`ImageStorage.swift:185-217`) read pre-v2 data. Disproven by: the only test that names legacy specifically is `testMigrateToV2ReturnsNilForV2Format` (CryptoServiceTests.swift:88-91) which confirms v2 input is **not** mistakenly treated as legacy. No test crafts an authentic pre-v2 blob (16-byte IV + ciphertext + 32-byte HMAC) and round-trips through `decryptLegacy`. The path is reachable in production via `loadItems` migration (ClipboardStore.swift:208-224) and `ImageStorage.loadImage` fallback (172-179), so any regression would surface on an existing user's next launch with legacy data. **Test gap; should be added as `testLegacyDecryptRoundTrip` for coverage hardening but not a blocker for v2.2.4.**
- **RS-7 — `tccutil`/`cfprefsd` UserDefaults flush on quit:** related to RS-1; `flushSave` in `ClipboardStore.deinit` (`ClipboardStore.swift:154-158`) calls `saveItems()` → `backend.save` → async dispatch. If `NSApp.terminate` follows immediately, the dispatched block may not fire. Disproven by: `deinit` is only invoked when the last `ClipboardStore.shared` reference is released, which on app quit happens after the typical pre-teardown cycles; `cfprefsd` flushes UserDefaults during the brief window of normal app shutdown. **Pre-existing design choice; not a defect.**
